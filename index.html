Guía verificada · specify 0.11.8

Especifica el qué. Claude construye el cómo.

Desarrollo guiado por especificaciones (Spec-Driven Development): defines el qué y el porqué, y Claude Code genera el cómo a partir de specs ejecutables, en pasos refinados — no en un solo prompt.

  2. specify
  3. clarify

  6. analyze

1 Qué es Spec-Kit

Spec-Kit es un toolkit open-source de GitHub para Spec-Driven Development (SDD). Invierte el flujo tradicional: en vez de tratar la spec como andamio desechable, la convierte en especificación ejecutable. El agente de IA (Claude Code) implementa directamente a partir de specs detalladas.

Filosofía central: intent-driven development. La especificación define el qué antes que el cómo, con refinamiento en varios pasos en lugar de prompts de un solo tiro. Objetivo: dejar de "vibe-codear" cada pieza y obtener outcomes predecibles.

2 Instalación

Requisitos: Python 3.11+, Git y uv (o pipx). El CLI se llama specify.

# Instalar el CLI specify (usa el último release de GitHub)
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git
Versión

Vas a usar el último release de GitHub. Sin tag, el comando de arriba instala HEAD del repo. Para fijar una versión concreta añade @vX.Y.Z con un tag de la página de Releases. Tras instalar, confirma con specify self check.

Auto-gestión del CLI

specify self check                 # ¿hay updates?
specify self upgrade               # actualizar al último
specify self upgrade --tag vX.Y.Z   # fijar versión concreta

3 Inicializar el proyecto para Claude Code

# Proyecto nuevo
specify init mi-proyecto --integration claude
cd mi-proyecto

# O inicializar en el directorio actual (--here equivale a usar '.')
specify init . --integration claude --force
Flag correcto (verificado en specify 0.11.8)

El flag para Claude Code es --integration claude (no --ai). En sesiones no interactivas, sin flag, el CLI usa Copilot por defecto — así que para Claude debes pasarlo explícito. Si Claude Code no está en el PATH, añade --ignore-agent-tools para saltar la comprobación. init es offline: los templates van empaquetados en el CLI, así que coinciden con tu versión instalada.

Esto instala los comandos como skills en .claude/skills/ (p. ej. speckit-constitution) y crea el andamiaje en .specify/.

Nombres de comando: guion, no punto

Con la integración Claude Code, los comandos se invocan con guion: /speckit-specify, /speckit-plan, etc. La forma con punto que ves en docs (speckit.specify) corresponde a otras integraciones (Copilot y similares). Esta guía usa la forma de Claude Code: guion.

5 Quick Start: flujo de trabajo completo ★ lo esencial

Ejemplo end-to-end: construir "LinkSnip", un acortador de URLs con dashboard (stack JS/TS). Cada paso es un slash command dentro de Claude Code, en este orden.

Cómo leerlo

Todo lo que empieza por /speckit- se teclea como slash command dentro de Claude Code (no en la terminal). El texto entre comillas es el prompt que escribes junto al comando. Los bloques de terminal (npm, npx…) van en tu shell.

  1. /speckit-constitution ·fija los principios

    Establece las reglas que Claude respetará en todo el proyecto. Cuanto más concretas y verificables, menos deriva. Ejemplo de constitution endurecida contra sobre-ingeniería:

    Ver la constitution completa 
    /speckit-constitution Principios de ingeniería para LinkSnip. Son vinculantes
y prevalecen sobre cualquier "mejor práctica" que añada complejidad no pedida.

1. TEST-FIRST — Ninguna línea de implementación sin un test que falle antes.
   Cobertura mínima 80%. Un test debe fallar si la lógica de negocio cambia.
2. LIBRARY-FIRST — Cada feature nace como módulo independiente y testeable de
   forma aislada antes de integrarse.
3. TIPADO ESTRICTO — TypeScript en modo strict. Prohibido 'any' y los tipos
   implícitos. Los contratos se modelan con tipos, no con comentarios.
4. MANEJO DE ERRORES EXPLÍCITO — Todo camino de error se identifica y cubre
   antes de codear el camino feliz. Nada de fallos silenciosos.
5. SIMPLICIDAD SOBRE INGENIO — Prioriza legibilidad y baja complejidad
   cognitiva. Si 50 líneas resuelven lo que escribiste en 200, reescribe.
6. YAGNI — ANTI SOBRE-INGENIERÍA (regla dura):
   - No implementes nada que no trace a un requisito explícito del PRD o a un
     criterio de aceptación de la spec ACTUAL.
   - Prohibido: abstracciones "por si acaso", capas de configuración no pedidas,
     parámetros/opciones especulativos, patrones de diseño sin caso de uso
     presente, dependencias para problemas que aún no existen, soporte para
     escalas o casos que nadie ha solicitado.
   - Construye la solución más simple que satisfaga el requisito de HOY. La
     generalización llega con el SEGUNDO caso real, no antes.
7. ZERO BREAKING CHANGES — Ningún cambio incompatible en la API pública sin
   versionar.

GOVERNANCE
- Antes de implementar, el agente debe justificar cada componente, abstracción
  o dependencia citando el requisito concreto que lo motiva. Si no puede, lo
  elimina.
- En /speckit-plan y /speckit-analyze, marca cualquier elemento que viole YAGNI
  como [OVER-ENGINEERING] y propón la versión mínima.
- Estos principios ganan a cualquier sugerencia que añada complejidad no
  solicitada.

    constitution.md recoge estos principios y el agente los cita al planear.

  2. /speckit-specify ·define el QUÉ (sin stack)

    Requisitos y user stories. Describe necesidades de usuario, no tecnología.

    /speckit-specify Una app web donde un usuario pega una URL larga y recibe un
enlace corto. Puede ver una lista de sus enlaces con el conteo de clics. Los
enlaces caducan a los 30 días. Página pública de redirección. Sin login en v1.

    spec.md describe qué/por qué; cero menciones de framework o base de datos.

  3. /speckit-clarify ·resuelve ambigüedades

    Antes de planear: cada ambigüedad resuelta aquí es menos rework después, así que no te lo saltes.

    /speckit-clarify

    No quedan marcadores [NEEDS CLARIFICATION] sin resolver en spec.md.

  4. /speckit-plan ·el CÓMO (deja que la IA recomiende el stack)

    En vez de dictar tú la tecnología, pide a Claude que proponga las mejores opciones para los requisitos y restricciones, con trade-offs y una recomendación por defecto. Tú decides después.

    /speckit-plan No fijes el stack todavía. Primero, basándote en la spec y la
constitution, recomienda la mejor arquitectura para LinkSnip bajo estas
restricciones: equipo pequeño con TypeScript, despliegue barato/serverless,
lecturas de redirección de baja latencia, sin login en v1, mantenimiento simple.

Para cada decisión técnica (lenguaje/runtime, framework web, persistencia,
generación de IDs cortos, framework de tests, hosting) preséntame:
  - 2-3 opciones viables con pros/contras en una frase
  - tu recomendación marcada como [RECOMENDADO] y por qué gana aquí
  - qué requisito concreto de la spec la justifica (si no traza, no la propongas)

Optimiza para simplicidad y baja complejidad cognitiva (constitution §5 y §6).
Marca cualquier alternativa que solo aporte flexibilidad especulativa como
[OVER-ENGINEERING]. Espera mi OK a las recomendaciones antes de escribir plan.md.

    El plan ofrece opciones con trade-offs y un [RECOMENDADO] justificado por requisito; nada de stack impuesto a ciegas.

    Por qué así

    Dejar que la IA proponga y justifique convierte el plan en una decisión informada, no en un capricho del prompt. El gate [RECOMENDADO] / [OVER-ENGINEERING] conecta con la GOVERNANCE de tu constitution: cada pieza debe trazar a un requisito. Si ya tienes el stack decidido, sáltate esto y díctalo — pero entonces pierdes la auditoría de alternativas.

  5. /speckit-tasks ·desglose ordenado

    Genera tasks.md en fases (Setup → Foundational → User Stories → Polish), con cada task etiquetada [ID] [P] [US#]. No es una lista plana — cada user story es un incremento entregable. Ver cómo se ejecuta en §5b.

    /speckit-tasks

    tasks.md organizado por fases/stories, con dependencias y marcas [P] correctas.

  6. /speckit-checklist ·checklist de calidad opcional

    Genera un checklist de validación a medida contra la spec.

    /speckit-checklist Genera un checklist de aceptación enfocado en seguridad
de redirección, validación de URLs y caducidad de enlaces.

    Checklist cubre los riesgos clave de la feature.

  7. /speckit-analyze ·consistencia cruzada

    Valida que spec ↔ plan ↔ tasks no se contradigan antes de escribir código.

    /speckit-analyze

    Cero inconsistencias reportadas entre artefactos.

  8. /speckit-implement ·construir

    Un solo comando ejecuta TODAS las tasks de tasks.md, fase a fase y en orden de dependencias (no relanzas por task). Detalle completo del modelo de ejecución en §5b. Instala antes el tooling local.

    # Terminal: tener Node/herramientas listas antes
node -v # confirma toolchain
    # Todo de una pasada:
/speckit-implement

# …o incremental: scópalo a una user story y para en el checkpoint
/speckit-implement Solo la Phase 3 (User Story 1 / MVP); detente al terminarla.

    La app arranca, los tests pasan; devuelve a Claude cualquier error de runtime para que lo resuelva.

  9. /speckit-taskstoissues opcional · tracking en GitHub

    Convierte las tareas en issues de GitHub. Necesita la CLI gh instalada y autenticada.

    /speckit-taskstoissues

    Issues creados en el repo, uno por tarea.

5b Cómo se ejecutan las tasks (¿una a una o todas?)

Duda típica: "si mi app son 5 tasks, al lanzar la implementación, ¿se ejecutan las 5 o solo la primera?". Respuesta corta, verificada leyendo el skill speckit-implement de specify 0.11.8:

Respuesta directa

Un solo /speckit-implement ejecuta TODAS las tasks pendientes de tasks.md en una pasada. Con 5 tasks, las hace las 5 — no relanzas el comando por cada una. Internamente es un bucle automático: va fase a fase, en orden de dependencias, marca cada task como [X] al terminar y te reporta el progreso, en vez de obligarte a lanzar la 1ª y luego la 2ª a mano.

1 · tasks.md no es una lista plana: está por fases y user stories

/speckit-tasks no genera "5 tareas sueltas". Las organiza en fases, y las de funcionalidad se agrupan por user story (cada story = un incremento entregable y testeable por separado):

Phase 1: Setup              # estructura, deps, config
Phase 2: Foundational       # infra base — BLOQUEA todas las stories
Phase 3: User Story 1 (P1)  # 🎯 MVP
Phase 4: User Story 2 (P2)
Phase 5: User Story 3 (P3)
Phase N: Polish             # docs, perf, hardening

Cada línea de task tiene el formato [ID] [P?] [Story] descripción + ruta:

- [ ] T012 [P] [US1] Crear modelo Link en src/models/link.ts
- [ ] T014    [US1] Implementar LinkService (depende de T012, T013)
  • T012 = ID secuencial.
  • [P] = paralelizable (archivos distintos, sin dependencias). Sin [P] = secuencial.
  • [US1] = a qué user story pertenece (trazabilidad).
  • [ ] → [X] cuando se completa.

2 · Qué hace /speckit-implement por dentro

ComportamientoDetalle
AlcanceProcesa toda la tasks.md en una invocación
OrdenFase a fase: completa una fase antes de pasar a la siguiente
DependenciasSecuenciales en orden; las [P] pueden ir juntas; mismo archivo → secuencial
TDDSi hay tests, los escribe y los hace fallar antes de implementar
ProgresoReporta tras cada task y marca [X] en tasks.md
ErroresSe detiene si falla una task no-paralela; en [P] sigue con las OK y reporta las fallidas
Gate de checklistsSi hay checklists incompletos, PARA y te pregunta si continuar (sí/no)

3 · Cómo ir poco a poco (control fino / entrega incremental)

Si no quieres soltar las 5 de golpe, tienes dos vías reales:

  1. Filtro por argumento. El skill acepta una guía/filtro opcional (argument-hint: "implementation guidance or task filter"). Scópalo a una fase o story: 
    /speckit-implement Implementa solo la Phase 3 (User Story 1 / MVP) y
detente en el checkpoint. No toques US2 ni US3 todavía.
    Validas, y luego relanzas para la siguiente story.
  2. MVP-first por diseño. Como tasks.md ya separa stories independientes, el patrón recomendado es: Setup + Foundational → US1 → STOP & validar → demo → US2 → … Cada story añade valor sin romper la anterior.
Modelo mental

1 feature → 1 spec.md → 1 plan.md → 1 tasks.md (N tasks en fases/stories).   1 /speckit-implement = ejecuta todas las pendientes en orden, no una por comando.   ¿Quieres granularidad? Pásale un filtro de fase/story y valida en cada checkpoint.

Ojo: una feature a la vez

Este flujo es por feature. /speckit-specify crea una carpeta specs/[feature]/ nueva (y su rama). Si tu app tiene varias features grandes, repites el ciclo specify→plan→tasks→implement por cada una, no metes todo en una sola spec gigante.

7 Reglas por paso (lo que separa buen output de malo)

Constitution

Define principios antes de todo. Luego pide explícitamente a Claude que respete la constitution al tomar decisiones de arquitectura. Es la capa que mantiene coherencia entre features.

Specify

Explícito en el qué y el porqué. Nada de "usar Postgres/React" aquí: si mezclas stack contaminas la spec y atas decisiones que aún no toca tomar.

Clarify

El paso con mayor ROI. Cada ambigüedad resuelta antes de planear evita rework caro aguas abajo. No lo saltes nunca.

Plan

Mejor patrón: pide a Claude que recomiende opciones con trade-offs y marque su preferida como [RECOMENDADO], en vez de imponerle tú el stack a ciegas. Luego pídele que audite el plan en busca de sobre-ingeniería ([OVER-ENGINEERING]) y valida antes de implementar. Cada pieza debe trazar a un requisito.

Implement

Instala las herramientas locales antes (Node, npm, etc.). Tras correr: prueba la app y devuelve los errores de runtime a Claude para que los resuelva — es parte del loop, no un extra.

8 Presets, bundles y workflows (mecánica)

  • Extensiones: añaden capacidades nuevas (comandos + hooks). CLI: specify extension.
  • Presets: modifican el comportamiento de workflows existentes. CLI: specify preset.
  • Bundles: paquetes curados de extensiones + presets para un rol (PM, business analyst, developer). CLI: specify bundle.
  • Workflows: automatizaciones ejecutables. CLI: specify workflow.
  • Todo resuelve en runtime con orden de prioridad (menor número = más precedencia). Los overrides locales del proyecto van en .specify/templates/overrides/.
# Aplicar un preset al inicializar
specify init mi-proyecto --integration claude --preset <nombre-preset>

# Gestión (subcomandos análogos a extension: list/add/remove/...)
specify preset --help
specify bundle --help
specify workflow --help
Verificado

Los grupos de comandos extension, preset, bundle, workflow, integration existen en specify 0.11.8 (vistos en specify --help). Para los subcomandos exactos de preset/bundle, corre su --help en tu proyecto.

9 Tips y gotchas

  • El orden importa. constitution → specify → clarify → plan → tasks → analyze → implement. Saltarte clarify o analyze es donde aparece el rework.
  • Separa el "qué" del "cómo". Stack solo en /speckit-plan, nunca en /speckit-specify.
  • Audita el plan. Pide explícitamente a Claude detectar sobre-ingeniería antes de implementar.
  • Devuelve errores. Tras implementar, corre la app, copia errores de runtime al chat y deja que Claude itere.
  • Prepara el toolchain. Node/npm/herramientas instaladas antes de /speckit-implement.
  • Versiona .specify/ y specs/. Son artefactos de primera clase, no temporales.
  • Nombres de comando según integración. Claude Code los instala como skills con guion: /speckit-plan. Otras integraciones (Copilot, etc.) usan la forma con punto /speckit.plan.
  • Mide siempre. "Parece que funciona" no es señal; cobertura ≥90% y lint en cero sí lo son.
